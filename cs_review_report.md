# yah.mobile チャットサポート — カスタマーサポート専門家レビュー

**レビュー日**: 2026年6月26日  
**対象**: yah-chat-webdev（chat.yah.mobi）  
**レビュアー視点**: カスタマーサポートセンター スペシャリスト

---

## 総合評価

**方向性は正しい。ただし、実装の一貫性に重大な欠陥が3点ある。**

ボットファースト・フォーム誘導型のアーキテクチャは、yah.mobileのサービス特性（問い合わせパターンが限定的・デジタル商品・多言語対応）に非常に適合している。99.9%のAI解決率という目標も現実的に達成可能な水準だ。しかし現状では、訪問者が**どのルートからチャットを開始するかによって、まったく異なる体験**を受けることになっており、これはブランドの信頼性を損なう。

---

## 重大な問題（即時対応推奨）

### 問題1: `/chat/:sessionId` ルートが旧モデルのまま放置されている

**現象**: `ChatRoom.tsx`（`/chat/:sessionId`）には以下が残存している。
- 「オペレーターに繋ぐ」ボタンが入力エリア上部に**常時表示**
- 「Connect to operator」バナーが表示される`shouldEscalate`ロジック
- `escalation_suggested`のSocket.ioリスナー

**影響**: `ChatStart.tsx`（`/chat`）からチャットを開始した訪問者は、WidgetChatとは異なり**オペレーター引き継ぎボタンが常に見える**。yah.mobiのフッターやリンクから`/chat`に直接誘導されているケースがあれば、ポリシー変更が完全に無効化される。

**推奨対応**: `ChatRoom.tsx`の`shouldEscalate`関連UIを削除し、`shouldRedirectToForm`バナーに統一する。または`/chat`ルート自体を廃止してWidgetChatに一本化する。

---

### 問題2: フォーム誘導の条件が実質的に機能しない可能性がある

**現象**: フォーム誘導の発火条件は「**AIメッセージ数 ≥ 10 かつ 未解決シグナルキーワードを含む**」の両方が必要。

```typescript
const shouldRedirectToForm = aiMessageCount >= 10 && userSignalsUnresolved;
```

**問題点**:
1. **キーワード依存の脆弱性**: 「接続できません」「動きません」などの自然な表現がキーワードリストに含まれていないと、10回以上やり取りしても誘導されない。例えば日本語の「繋がらない」「表示されない」はリストにない。
2. **10回という閾値の高さ**: 実際のサポートでは、5〜6回のやり取りで解決しない場合はエスカレーションが標準的。10回は訪問者にとってストレスが高い。
3. **AND条件の問題**: キーワードを含まない「もうやめます」「諦めます」という離脱シグナルではフォーム誘導が発火しない。

**推奨対応**: 
- 条件を「10回以上 **または** 未解決シグナル」（OR条件）に変更する
- 日本語キーワードリストに「繋がらない」「表示されない」「できません」「わかりません」を追加
- 離脱シグナル（「やめます」「諦め」）も追加する

---

### 問題3: HERO KPIの計測方法が実態を反映していない

**現象**: `aiResolvedRate`はアンケート回答（`resolved: "yes"`）を分母に計算されている。

```typescript
const aiResolvedRate = aiSurveys.length > 0
  ? Math.round((aiSurveys.filter((s) => s.resolved === "yes").length / aiSurveys.length) * 100)
  : null;
```

**問題点**:
1. **アンケート回答率が低い場合、数値が歪む**: チャット終了後のアンケートに回答するのは一般的に20〜40%程度。回答しない訪問者（解決した可能性が高い）が除外されるため、実際の解決率より**低く出る**傾向がある。
2. **アンケート未提出 = 未解決と見なされる**: 解決して満足した訪問者がアンケートをスキップすると、分母に入らない。
3. **Data Analysisのサブテキスト「Based on post-chat surveys」**は正確だが、Yoshiさんがこの数値を「全体の解決率」と誤解するリスクがある。

**推奨対応**: 
- 補助指標として「セッション終了率（自然終了 / 全セッション）」を追加し、アンケートベースの数値と並列表示する
- アンケート回答率（`surveyCount / endedSessions`）もKPIに追加して数値の信頼性を可視化する

## 中程度の問題（近日対応推奨）

### 問題4: ヘッダーのサブテキストが旧ポリシーを示唆している

**現象**: WidgetChatのチャット中ヘッダーに「AI + Operator」と表示されている（line 401）。

```tsx
{stage === "chat" ? "AI + Operator" : "Chat Support"}
```

**影響**: 訪問者に「オペレーターがいる」という期待を持たせる。ボットファースト方針と矛盾する。

**推奨対応**: 「AI Support」または「24/7 AI Support」に変更する。

---

### 問題5: コメント内の古い記述が残っている

**現象**: WidgetChat.tsx line 524 に「after 3 attempts」というコメントが残っている（実際のロジックは10回）。

```tsx
{/* Contact form redirect banner — shown when AI cannot resolve after 3 attempts */}
```

**影響**: 開発者・レビュアーが誤解する。コードの信頼性を下げる。

**推奨対応**: コメントを「after 10 attempts」に修正する。

---

### 問題6: 会話履歴の参照が直近10件のみ

**現象**: AIへの会話履歴は直近10件のみ渡している。

```typescript
const conversationHistory = history.slice(-10).map(...)
```

**影響**: 長い会話（10往復以上）で文脈が失われ、AIが同じ提案を繰り返す。これが「解決できない」と感じさせる主因になりうる。

**推奨対応**: 直近15〜20件に拡張する（コスト増は軽微）。または会話要約をシステムプロンプトに組み込む。

---

### 問題7: 返金ポリシーの例外ケース処理が不完全

**現象**: システムプロンプトに例外ケース（システム障害・二重請求・不正利用）への言及があるが、AIがこれらを判断する情報（注文ID・決済記録）を持っていない。

**影響**: AIが「例外ケースに該当する可能性があります。フォームからご連絡ください」と案内できるが、訪問者が「例外に該当するかどうか」を自己判断して問い合わせてくるため、フォームへの問い合わせ数が増える可能性がある。

**推奨対応**: 例外ケースの判断基準を明確にし、「以下のいずれかに該当する場合のみフォームへ」という形で訪問者が自己選別できるようにする。

## まとめ

**ボットファースト戦略の方向性は正しく、技術基盤も整っている。**  
最大のリスクは「`/chat`ルートと`/widget-chat`ルートで体験が分断されていること」であり、これを解消するだけで一貫したボットファースト体験が実現する。フォーム誘導の条件改善（AND→OR）も合わせて対応することで、実際の解決率向上が期待できる。
