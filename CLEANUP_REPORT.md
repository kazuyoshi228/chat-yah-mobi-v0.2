### 1.4 セキュリティ強化 — レート制限の実装

`@upstash/ratelimit` を使用した3層のレート制限を導入しました。

| 制限対象 | 上限 | ウィンドウ | 目的 |
|---|---|---|---|
| **セッション作成** | 5回 | 1時間 / visitorId | ボットによる大量セッション生成を防止 |
| **メッセージ送信** | 30回 | 5分 / sessionId | 短時間の連続送信（スパム）を防止 |
| **QR再送** | 3回 | 30分 / email | QR再送の悪用を防止 |

### 1.5 チャット悪用対策 — メッセージ数上限

| 制限 | 値 | 効果 |
|---|---|---|
| セッションあたりメッセージ上限 | 50メッセージ | ダラダラ会話・API消費攻撃を防止 |
| 上限到達時 | エラーメッセージ表示 + フォーム誘導 | ユーザーを適切な解決手段へ誘導 |

## 2. 追加改善提案（未実施）

### 2.1 セキュリティ

| 提案 | 優先度 | 理由 |
|---|---|---|
| **Helmet.js 導入** | 高 | HTTP セキュリティヘッダー（CSP, X-Frame-Options, HSTS等）を一括設定。現在はデフォルトのExpressヘッダーのみ |
| **入力サニタイゼーション強化** | 中 | メッセージ内容の XSS ペイロード検出。現在は Zod バリデーションのみ |
| **Webhook 署名検証の厳格化** | 中 | 現在の `WEBHOOK_SECRET` ヘッダー比較に加え、HMAC-SHA256 署名検証を推奨 |
| **セッション固定攻撃対策** | 低 | visitorId の生成ロジックをサーバーサイドに移行（現在はクライアント生成） |

### 2.2 チャット悪用対策

| 提案 | 優先度 | 理由 |
|---|---|---|
| **IP ベースのレート制限** | 高 | visitorId はクライアント側で変更可能。IP レベルの制限で回避を防止 |
| **CAPTCHA 統合** | 中 | セッション開始時に hCaptcha/Turnstile を要求。ボット攻撃の根本対策 |
| **コンテンツフィルタリング** | 中 | 繰り返し同一内容の送信を検出してブロック（反復攻撃対策） |
| **コスト上限アラート** | 高 | Anthropic API の月次コスト上限を設定し、超過時にAI応答を一時停止 |
| **無意味メッセージ検出** | 低 | 「あああ」「test」等の無意味入力が連続した場合にセッションを自動終了 |

### 2.3 ミニマル化・UX

| 提案 | 優先度 | 理由 |
|---|---|---|
| **HistoricalDocs.tsx の統合** | 低 | 内容が少なく、SSoT ページに統合可能 |
| **Improvements.tsx の統合** | 低 | 改善提案はこのレポートで代替可能。静的ページとしての価値が低い |
| **UserManuals.tsx の精査** | 低 | 内容が古くなっている可能性。最新の実装と照合が必要 |

## 3. 現在のセキュリティ状態サマリー

| カテゴリ | 状態 | 備考 |
|---|---|---|
| 認証（OAuth） | ✅ 安全 | Manus OAuth + JWT セッション |
